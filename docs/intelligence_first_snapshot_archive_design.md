# Intelligence-First Snapshot Archive Design

**Sprint:** 7H.1 — Operations readiness
**Status:** Design document. Snapshot archive is NOT implemented. This documents the design for a future implementation sprint.
**Classification:** Advisory/design document. No production code changed.
**Activation gate:** Must be approved by Amit before implementation. Implementation requires a dedicated sprint after the controlled activation sprint closes.

---

## 1. Purpose

The current publisher produces two primary outputs — `current_manifest.json` and `active_opportunity_universe.json` — and overwrites them in-place on each cycle. This means only the latest state is available at any time. A snapshot archive would retain a versioned history of every publisher output, enabling:

1. **Post-hoc audit**: reconstruct exactly what candidates were in the handoff universe at any point in time
2. **Divergence diagnosis**: compare candidate universe across cycles to detect drift, additions, or removals
3. **Training data enrichment**: link Apex decision records to the exact candidate set that was presented at entry time
4. **Rollback investigation**: after a rollback event, inspect the universe state that triggered the rollback trigger

This document defines the design. The implementation is deferred.

---

## 2. Archive Path

| Path | Contents |
|------|---------|
| `data/live/snapshots/` | Per-run subdirectories, one per publisher cycle |
| `data/live/snapshots/{run_id}/` | Files for that run: manifest, universe, and a snapshot index entry |
| `data/live/snapshots/snapshot_index.jsonl` | Append-only index: one line per run, with run_id, timestamp, candidate_count, expires_at |

**`run_id`** comes from the existing `run_id` field in `publisher_run_log.jsonl` (UUID generated by `_append_run_log()`). The run_id ties the snapshot, the run log entry, and the heartbeat together.

**Naming convention:**

```
data/live/snapshots/{run_id}/current_manifest.json
data/live/snapshots/{run_id}/active_opportunity_universe.json
```

No timestamp in the directory name — `run_id` is the stable identifier. The timestamp is in `snapshot_index.jsonl`.

---

## 3. What to Archive

| File | Archive? | Notes |
|------|---------|-------|
| `current_manifest.json` | **Yes** | Copy of the file as written in that cycle |
| `active_opportunity_universe.json` | **Yes** | Copy of the full universe for that cycle |
| `handoff_publisher_report.json` | **Yes** | Publisher's internal cycle report — diagnoses per-candidate |
| `handoff_publisher_observation_report.json` | **No** | Observer output; observer runs independently and post-dates the publisher cycle |
| `.fail_*.json` | **No** | Fail files are diagnostic, not part of the snapshot; they stay in `data/live/` |
| `publisher_run_log.jsonl` | **No** | The run log itself is the index; do not archive it into per-run subdirs |

**Archive timing:** The snapshot is written atomically after all four primary publisher outputs are written. If `_append_run_log()` succeeds, `_archive_snapshot()` runs next. If archiving fails, it is non-critical (warns, returns False, does not corrupt the manifest or run log).

---

## 4. What NOT to Archive

| Item | Reason not archived |
|------|-------------------|
| `data/reference/` files | Static; committed to repo; no change between cycles |
| Bot logs | Separate retention policy; not publisher artefacts |
| IBKR connection state | External system; not a publisher artefact |
| `advisory_runtime_log.jsonl` | Advisory system has its own retention policy |
| Any file from `data/intelligence/` | Intelligence worker outputs are not publisher artefacts |

---

## 5. Retention Policy for Snapshots

| Policy | Rule |
|--------|------|
| Retention period | Retain all snapshots from the controlled activation sprint |
| Post-sprint rotation | After activation sprint closes: prune to last 30 days of snapshots; compress older than 7 days |
| Max size estimate | Each snapshot ≈ 150 KB (manifest 10 KB + universe 140 KB). At 96 cycles/day: ~14 MB/day, ~430 MB/month. Manageable. |
| Snapshot index | `snapshot_index.jsonl` is never pruned — it is the lightweight index. Only the full snapshot directories are pruned |
| Pruning rule | Remove `data/live/snapshots/{run_id}/` directory; keep the `snapshot_index.jsonl` entry with a `pruned=true` flag |
| Git | Never committed to repo; on persistent volume only |

## 6. Implementation Gate and Future Sprint

This feature requires a dedicated implementation sprint. The gate to open that sprint:

| Gate condition | Status |
|----------------|--------|
| Controlled activation sprint is complete (success or rollback with root cause resolved) | Not yet reached |
| `publisher_run_log.jsonl` has accumulated ≥ 3 distinct UTC sessions of data | Met (as of Sprint 7G.1) |
| Amit approves snapshot archive sprint | Not yet approved |

**Why deferred:** The snapshot archive is not required for Phase 1 controlled activation. The run log and in-place manifest are sufficient for the activation sprint. Archiving adds I/O per cycle, increases disk footprint, and requires a new `_archive_snapshot()` function and a snapshot validator. This complexity is appropriate post-activation, not pre-activation.

**Implementation scope (for future sprint):**

1. Add `_archive_snapshot(run_id, now)` to `handoff_publisher.py` — copies manifest, universe, and report into `data/live/snapshots/{run_id}/`
2. Append one line to `data/live/snapshots/snapshot_index.jsonl` with `{run_id, utc_timestamp, candidate_count, expires_at, snapshot_written: true}`
3. Wire `_archive_snapshot()` as Step 7c in `run_publisher()`, after `_append_run_log()`
4. Add `validate_snapshot_index()` to `intelligence_schema_validator.py`
5. Add to `validate_all()` as file 41
6. Write 5 tests in `TestSnapshotArchive` group in `test_handoff_publisher.py`
7. Update `docs/intelligence_first_log_retention_and_diagnostics_policy.md` with snapshot retention details
8. Update this document status to "Implemented"
